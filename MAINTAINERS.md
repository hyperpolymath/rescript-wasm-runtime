## Decision Making

### Consensus Model
- **Minor changes**: Any maintainer can approve
- **Major changes**: 2 maintainer approvals required
- **Breaking changes**: All maintainers must approve
- **Disputes**: Lead maintainer has final say

### What Requires Consensus
- Breaking API changes
- New major features
- Architecture changes
- Dependency additions
- License changes
- Code of Conduct changes
- Maintainer additions/removals

## Becoming a Maintainer

### Criteria
1. **Sustained Contributions**: 6+ months of regular contributions
2. **Technical Excellence**: High-quality PRs, good judgment
3. **Community Engagement**: Helpful in issues, reviews
4. **Alignment**: Demonstrates project values and goals
5. **Commitment**: Available for ongoing maintenance

### Process
1. **Nomination**: Self-nominate or be nominated by maintainer
2. **Discussion**: Current maintainers discuss in private
3. **Vote**: Consensus required (all maintainers agree)
4. **Invitation**: Formal invitation sent
5. **Onboarding**: Access granted, added to MAINTAINERS.md
6. **Announcement**: Public announcement in project channels

### Trial Period
- New maintainers have a 3-month trial period
- Can be revoked if expectations not met
- After 3 months, becomes permanent

## Maintainer Expectations

### Time Commitment
- **Minimum**: 4 hours/month
- **Expected**: 8-16 hours/month
- **During releases**: 20+ hours/month

### Responsiveness
- **Issues**: Respond within 7 days
- **PRs**: Initial review within 7 days
- **Security**: Respond within 48 hours
- **Code of Conduct**: Respond within 24 hours

### Burnout Prevention
- Set realistic expectations
- Take breaks when needed
- Communicate availability
- Share responsibilities
- Ask for help
- Step down if needed (no judgment)
